## 1. Czym jest Claude Code?

Claude Code to **asystent kodowania działający w terminalu**, który daje modelowi AI (Claude) dostęp do Twojego systemu plików, komend i narzędzi.

**Kluczowa zasada:** Sam model językowy potrafi jedynie przyjmować i generować tekst. Nie umie czytać plików ani uruchamiać komend. Claude Code rozwiązuje to przez system **Tool Use** — model mówi „chcę przeczytać plik X", a asystent fizycznie to wykonuje na Twoim dysku.

### Przepływ działania

```
Ty → pytanie/zadanie
      ↓
Claude Code → dodaje instrukcje narzędzi do zapytania
      ↓
Model Claude → odpowiada np. "ReadFile: main.go"
      ↓
Claude Code → fizycznie czyta plik, zwraca zawartość
      ↓
Model Claude → analizuje i daje finalną odpowiedź
```

### Wbudowane narzędzia

| Narzędzie | Funkcja |
|-----------|---------|
| **Read** | Czytanie plików |
| **Write / Edit / MultiEdit** | Tworzenie i edycja kodu |
| **Grep** | Przeszukiwanie plików |
| **Bash** | Uruchamianie komend systemowych |
| **TodoWrite** | Zarządzanie listą zadań |

> **Moc Claude'a:** Inteligentnie łączy te narzędzia, samodzielnie ustalając plan działania — nie musisz go prowadzić za rękę.

## 2. Zarządzanie kontekstem

> ⚠️ **Zasada nr 1:** Zbyt dużo nieistotnego kontekstu **obniża** jakość odpowiedzi. Steruj uwagą modelu.

### Komenda `/init`

Odpalasz **raz** w nowym projekcie. Claude skanuje kod i generuje plik `CLAUDE.md` — permanentny System Prompt projektu.

### Trzy poziomy `CLAUDE.md`

| Plik | Zakres | Udostępnianie |
|------|--------|---------------|
| `CLAUDE.md` | Projekt | ✅ Commitowany do Git, współdzielony z zespołem |
| `CLAUDE.local.md` | Projekt | ❌ Prywatny, nie trafia do repo |
| `~/.claude/CLAUDE.md` | Globalny | Reguły dla **wszystkich** projektów na maszynie |

### Tryb pamięci `#`

Wpisujesz w terminalu:
```
# Pisz kod bez komentarzy
```
Claude **sam** dopisze tę regułę do Twojego `CLAUDE.md`.

### Odwołanie do pliku `@`

```
Jak działa system autoryzacji? @auth/login.ts
```
Claude automatycznie dołączy zawartość wskazanego pliku do kontekstu.

> **Pro tip:** Zaszyj kluczowe pliki w `CLAUDE.md`:
> *„Struktura bazy danych jest zdefiniowana w @prisma/schema.prisma. Sprawdzaj go przy każdej zmianie dotyczącej danych."*

## 3. Kontrola konwersacji

| Akcja | Skrót | Kiedy używać |
|-------|-------|-------------|
| **Przerwanie** | `Esc` | Claude idzie w złą stronę — ucinasz i przekierowujesz |
| **Cofnięcie** | `Esc` × 2 | Weszliście w ślepy zaułek — cofasz rozmowę do wcześniejszego punktu |
| **Kompresja** | `/compact` | Długa rozmowa, ale pełna cennej wiedzy — skompresuj, nie kasuj |
| **Reset** | `/clear` | Zupełnie nowe zadanie — czysta karta |

> **Combo ESC + #:** Ucinasz złą odpowiedź, dodajesz regułę pamięci i kontynuujesz — Claude już nigdy nie powtórzy tego błędu.

## 4. Tryby pracy

### Planning Mode *(Shift+Tab × 2)*

Dla zadań wymagających **szerokiego** zrozumienia projektu:
- Claude skanuje wiele plików
- Tworzy plan krok po kroku
- Czeka na Twoją akceptację przed zmianami

**Kiedy:** Zmiany w wielu plikach, refaktoryzacja architektury.

### Thinking Modes

Dla problemów wymagających **głębokiej** analizy:

| Tryb | Intensywność |
|------|-------------|
| `Think` | Podstawowa analiza |
| `Think more` | Rozszerzona |
| `Think a lot` | Komplementarna |
| `Think longer` | Przedłużona |
| `Ultrathink` | Maksymalna moc |

**Kiedy:** Skomplikowana logika, debugowanie, problemy algorytmiczne.

> **Można łączyć oba tryby**, ale zużywają dodatkowe tokeny.

### Screenshoty *(Ctrl+V)*

Wklejasz zrzut ekranu interfejsu, a Claude sam powiąże obraz z kodem. Mówisz „zmień kolor tego przycisku" — model wie, o który element chodzi.

## 5. Custom Commands (Własne komendy)

Twórz **własne makra** automatyzujące powtarzalne zadania.

### Tworzenie komendy

```
.claude/
  commands/
    audit.md       →  tworzy komendę /audit
    write_tests.md →  tworzy komendę /write_tests
```

### Przykład: `audit.md`

```markdown
1. Uruchom `npm audit`, aby znaleźć podatne pakiety
2. Uruchom `npm audit fix`, aby je załatać
3. Uruchom testy, żeby upewnić się, że nic się nie zepsuło
```

### Dynamiczne argumenty: `$ARGUMENTS`

Plik `write_tests.md`:
```markdown
Napisz testy dla: $ARGUMENTS

Konwencje testów:
* Framework: Vitest + React Testing Library
* Pliki testowe: __tests__/[nazwa].test.ts
* Pokrycie: happy path, edge case, error state
```

Użycie:
```
/write_tests hook/use-auth.ts
```

> Po dodaniu nowej komendy **zrestartuj** Claude Code.

## 6. MCP — Model Context Protocol

> 🔥 **Gamechanger kursu.** Pozwala Claude'owi wyjść poza terminal i podłączyć się do zewnętrznego świata.

### Czym jest MCP?

System **wtyczek** (serwerów), dzięki którym Claude zyskuje nowe narzędzia: sterowanie przeglądarką, dostęp do baz danych, integracja z chmurą.

### Instalacja serwera Playwright

```bash
claude mcp add playwright npx @playwright/mcp@latest
```

Claude zyskuje możliwość **sterowania prawdziwą przeglądarką**.

### Zarządzanie uprawnieniami

Plik `.claude/settings.local.json`:
```json
{
  "permissions": {
    "allow": ["mcp__playwright"],
    "deny": []
  }
}
```

> Uwaga na **podwójne podkreślniki** `mcp__playwright`.

### Przykład użycia (pętla samodoskonalenia)

```
„Otwórz localhost:3000, wygeneruj komponent, ZOBACZ efekt w przeglądarce,
 a potem popraw prompt generujący w @src/lib/prompts/generation.tsx,
 żeby następne komponenty wyglądały lepiej."
```

Claude sam uruchamia stronę → patrzy na wynik → poprawia instrukcje.

### Inne serwery MCP

- Bazy danych (SQLite, PostgreSQL)
- Testowanie API
- Integracje z chmurą (AWS, GCP)
- Automatyzacja narzędzi deweloperskich

## 7. Integracja z GitHub

Claude staje się **wirtualnym członkiem zespołu** wpiętym w repozytorium.

### Setup

```
/install-github-app
```

### Dwa główne workflow'e (GitHub Actions)

| Workflow | Działanie |
|----------|----------|
| **Mention** `@claude` | Oznaczeasz Claude'a w Issue/PR → analizuje problem i odpowiada z rozwiązaniem |
| **Auto Code Review** | Każdy nowy Pull Request → automatyczny audyt z raportem |

### Konfiguracja workflow'u

```yaml
# Przygotowanie środowiska
- name: Project Setup
  run: |
    npm run setup
    npm run dev:daemon

# Instrukcje kontekstowe
custom_instructions: |
  Projekt jest już skonfigurowany. Serwer działa na localhost:3000.

# Serwery MCP
mcp_config: |
  {
    "mcpServers": {
      "playwright": {
        "command": "npx",
        "args": ["@playwright/mcp@latest"]
      }
    }
  }

# Uprawnienia — KONIECZNE wylistowanie KAŻDEGO narzędzia
allowed_tools: "Bash(npm:*),mcp__playwright__browser_click,..."
```

> ⚠️ W GitHub Actions **nie ma skrótów** — każde narzędzie MCP musi być wymienione z osobna.

## 8. Hooks (Zaczepy)

Automatyczne **wyzwalacze akcji** — wpinajesz własne skrypty w działanie Claude'a.

### Dwa typy hooków

| Typ | Moment | Możliwości |
|-----|--------|-----------|
| **PreToolUse** | PRZED akcją | ✅ Może **zablokować** operację |
| **PostToolUse** | PO akcji | Reakcja: formatowanie, testy, feedback |

### Konfiguracja

Pliki hooków:
- `~/.claude/settings.json` — globalnie
- `.claude/settings.json` — projekt (zespół)
- `.claude/settings.local.json` — prywatne

### Przykład: Ochrona pliku `.env`

Konfiguracja (JSON):
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Grep",
        "hooks": [
          {
            "type": "command",
            "command": "node ./hooks/read_hook.js"
          }
        ]
      }
    ]
  }
}
```

Skrypt `read_hook.js`:
```javascript
async function main() {
  const chunks = [];
  for await (const chunk of process.stdin) {
    chunks.push(chunk);
  }
  const toolArgs = JSON.parse(Buffer.concat(chunks).toString());
  const readPath = toolArgs.tool_input?.file_path || "";

  if (readPath.includes('.env')) {
    console.error("Dostęp do pliku .env zablokowany!");
    process.exit(2);  // Kod 2 = BLOKADA
  }
}
main();
```

### Kody wyjścia

| Kod | Znaczenie |
|-----|-----------|
| `0` | ✅ Pozwól na operację |
| `2` | 🛑 Zablokuj operację (tylko PreToolUse) |

### Praktyczne zastosowania hooków

| Hook | Efekt |
|------|-------|
| **TypeScript Compiler** (Post) | Po edycji → automatyczne `tsc --noEmit` → błędy wracają do Claude'a |
| **AI weryfikujące AI** (Post) | Nowy kod w `/queries` → druga instancja Claude'a sprawdza, czy nie ma duplikatów |
| **Formatter** (Post) | Po zapisie pliku → automatyczne Prettier/ESLint |
| **Testy** (Post) | Po edycji → automatyczne uruchomienie testów |

### Dodatkowe typy hooków

| Event | Kiedy się odpala |
|-------|-----------------|
| `Notification` | Claude potrzebuje zgody lub jest bezczynny 60s |
| `Stop` | Claude skończył odpowiadać |
| `SubagentStop` | Pod-agent zakończył zadanie |
| `PreCompact` | Przed kompresją konwersacji |
| `UserPromptSubmit` | Użytkownik wysłał prompt |
| `SessionStart` / `SessionEnd` | Początek/koniec sesji |

### Pro tip: Debugowanie hooków

Podepnij „logujący" hook, który zapisze dane wejściowe do pliku:
```json
{
  "matcher": "*",
  "hooks": [{ "type": "command", "command": "jq . > hook-log.json" }]
}
```
Otwierasz `hook-log.json` i widzisz dokładną strukturę danych.

## 9. Claude Code SDK

Claude Code jako **biblioteka programistyczna** — wywoływany z Twoich skryptów (TypeScript/Python).

```typescript
import { query } from "@anthropic-ai/claude-code";

const prompt = "Znajdź duplikaty zapytań w ./src/queries";

for await (const message of query({ prompt })) {
  console.log(JSON.stringify(message, null, 2));
}
```

### Kluczowe cechy

- **Read-only domyślnie** — SDK nie może modyfikować kodu, chyba że jawnie zezwolisz
- Włączenie zapisu: `allowedTools: ["Edit"]`
- Ten sam silnik co wersja terminalowa
- Idealny do **pipeline'ów CI/CD**, automatycznych code review, generowania dokumentacji
